# Wind Power Forecasting

Прогноз почасовой выработки ВЭС мощностью 90.09 МВт: 26 турбин Siemens Gamesa
по 3.465 МВт. Основная цель проекта - построить `submission.csv` для периода
`2026-01-01 00:00:00` ... `2026-03-31 23:00:00`.

## Данные

```text
data/
  dataset/
    train_dataset.csv     # 2022-01-01 ... 2025-12-31, признаки + таргет
    valid_features.csv    # 2026-01-01 ... 2026-03-31, только признаки
```

Целевая колонка:

```text
Выработка. Результирующий расчет
```

Валидационный период содержит только значения ремонта `3` и `4` в колонке
`Кол-во_ВЭУ_в_ремонте`. Это учитывается при подборе весов ансамбля.

## Установка

Рекомендуется создать новое виртуальное окружение, потому что сохраненное
локальное `.venv` может быть непереносимым между машинами.

```bash
python -m venv .venv
.venv\Scripts\activate
python -m pip install --upgrade pip
pip install -r requirements.txt
```

## Запуск

```bash
python solution.py
```

На выходе создаются:

- `submission.csv` - одна колонка `Выработка. Результирующий расчет`, без индекса.
- `report.json` - CV-метрики, веса ансамбля, выбранная стратегия и статистика сабмита.

Текущий сохраненный результат:

- строк в `submission.csv`: `2126`
- стратегия: `blend+bias`
- средний CV MAE выбранной стратегии: `8.3454 МВт`
- диапазон предсказаний: `0.0` ... `78.59 МВт`

## Метрика

Метрика задана организаторами:

```text
x_i = |R_ki - E_fi| / N_уст * 100 %,
итог = mean_i(x_i)
```

- `R_ki` — фактическая выработка ВЭС в час `i`, МВт.
- `E_fi` — прогнозная выработка ВЭС в час `i`, МВт.
- `N_уст` — установленная мощность ВЭС, `90.09 МВт`.

То есть метрика — это MAE в МВт, делённый на установленную мощность и выраженный
в процентах. В коде это `metric_percent` и `mae_mw` в `report.json`.

## Основной подход

Боевой пайплайн находится в `solution.py`.

Активный ансамбль:

- **CatBoost** с MAE loss и категориальными бинами.
- **LightGBM** с `regression_l1`.
- **XGBoost** с `reg:absoluteerror`.
- **TCN** из `nn_model.py`, обучаемая на 24-часовых окнах метеопризнаков.

Веса ансамбля подбираются по out-of-fold предсказаниям на трех сезонных
holdout-фолдах: Q1/2023, Q1/2024 и Q1/2025. Затем сравниваются несколько
калибровок: простой blend, помесячный bias, regime-bias по бинам скорости ветра
и ремонту, residual-correction, isotonic calibration и две stacking-модели.
`solution.py` сам переоценит стратегии и выберет лучшую по CV.

Для быстрого smoke-прогона можно использовать:

```bash
set WIND_SMOKE_TEST=1
python solution.py
```

В этом режиме используются короткие модели и два Q1-фолда; результат пишется в
`report_smoke.json` и `submission_smoke.csv`, не затирая основной сабмит.

## Признаки

- Циклические признаки часа, месяца и дня года.
- Степени и бины скоростей ветра на 10/80/120/180 м.
- Взвешенная скорость `0.68 * V80 + 0.24 * V120 + 0.08 * gusts`.
- Sin/cos и секторные признаки направлений ветра.
- Разности и отношения скоростей на разных высотах, оценка wind shear.
- Количество доступных турбин и доступная мощность.
- Физическая кривая мощности Siemens Gamesa с cut-in 3 м/с, rated 12 м/с,
  cut-out 25 м/с.
- Плотность воздуха и density-adjusted power proxies.
- Rolling mean/std и diff-признаки только по метеоданным.
- Дополнительные Open-Meteo признаки из `openmeteo_extra.csv`, если файл есть.

## Эксперименты

- `optuna_boosters_multifold.py` - подбор CatBoost/LightGBM на Q1-фолдах.
- `optuna_tune.py` - старый подбор на одном Q1/2025 фолде.
- `optuna_nn.py`, `optuna_nn_multifold.py` - подбор TCN/LSTM-параметров.
- `download_openmeteo.py` - загрузка дополнительных погодных признаков.
- `test_nbeats.py` - отдельный эксперимент с Darts/N-BEATS; для него может
  потребоваться установить `darts` отдельно.

`report.json` также содержит `regime_bias` и `error_analysis`: сравнение
калибровок по режимам ветра, разрезы ошибки по году фолда, месяцу, ремонту,
времени суток и бинам скорости ветра. Это основной ориентир для следующего
цикла улучшений.

## Замечания по воспроизводимости

- Все случайные сиды зафиксированы константой `SEED = 20260511` в начале
  `solution.py`. Она пробрасывается в CatBoost (`random_seed`), XGBoost
  (`random_state`), LightGBM (`random_state`) и TCN (`torch.manual_seed` +
  `np.random.seed`). Из-за этого при последовательном запуске на CPU прогон
  даёт одинаковый `submission.csv`.
- Версии зависимостей зафиксированы в `requirements.txt` (точные версии
  пакетов, не диапазоны).
- TCN обучается на CPU; при наличии CUDA результат может незначительно
  меняться из-за разной реализации свёрток на GPU.
- При запуске тюнинга через `optuna_*_multifold.py` сидируется TPESampler,
  поэтому Optuna-эксперименты повторяемы.
